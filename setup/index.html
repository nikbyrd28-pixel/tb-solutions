Three moving parts: static web pages (hosted on Vercel, code on GitHub), one Postgres database with logins (Supabase), and an automation server (n8n on Hostinger). No servers to maintain, no frameworks to update — every page is a single self-contained HTML file.

4 · The databases

Base project (the brain)

leads your sales pipeline · clients your book of business · client_leads every lead captured for any client · prospects signups · intakes project intakes · tickets support · reviews testimonials (approved → homepage) · marketing_assets AI library · client_profiles brand voices · posts social · updates client updates · pageviews analytics · client_errors JS error log · activities automation log · storage: uploads (public), backups (private)

The security pattern (row-level security)

Every table has RLS ON. Three recurring policy shapes: public-can-insert (forms write, never read) · owner-only (auth.uid() = user_id) · admin-only (email check). Clients see their own leads via a normalized business↔client name match. Automations in the database itself: a signup triggers prospect + lead creation; marking a lead Won auto-creates the client in HQ.

6 · How to change anything (the 3-step loop)

1
Edit the file on GitHub. github.com → tb-solutions repo → open the file (see §3 for which file is which page) → pencil icon → make your change. Works from your phone.
2
Commit. Scroll down → "Commit changes". That's the save button.
3
Wait ~30 seconds. Vercel deploys automatically. Refresh tbsol.net — it's live. Broke something? GitHub → History on that file → restore the previous version the same way.

Database changes happen in Supabase → SQL Editor. Rule: only ADD things (new tables, new columns) — never rename or delete columns the pages use.

7 · Cookbook — common tasks

Onboard a new client (1 min)
HQ → Clients → + New client (or mark their lead Won — it auto-creates). The popup gives their capture link, CRM, and portal — already working. Then: Marketing Engine → select them → write their brand profile → 🧰 Full campaign kit.
Put a review on the homepage
Send the client tbsol.net/review?b=TheirBusiness → HQ → Reviews → ✅ Approve. Live instantly.
Run ads for a client
Ad link = tbsol.net/capture/?c=slug&name=Business&utm_source=facebook. Leads land in their CRM + your Pipelines with the source tagged; Analytics shows per-source counts.
Change homepage copy / prices
Edit index.html via §6. Search for the exact text you see on screen to find the spot.
Add a new automation
Build it in n8n directly, or hand an AI one of the files in /n8n/ as the example format, commit the JSON to the repo's n8n folder, import from URL.
Restore from backup
Supabase → Storage → backups → download the latest JSON — it contains every table. Re-insert via SQL editor, or hand it plus this page to a developer.
Rotate keys (quarterly habit)
Gemini: aistudio.google.com/apikey → new key → update the n8n credential. n8n API key: Settings → n8n API. Gmail App Password: myaccount.google.com/apppasswords. GitHub tokens: Settings → Developer settings.
See errors users hit
Supabase → Table editor → client_errors — every JS error on every page, newest first.

8 · How this SaaS is built, step by step

The recipe — for rebuilding, for an IT person, or for building the next one:

1
Create a Supabase project. Database + user logins + file storage, free tier. Note the URL, anon key (public), and service_role key (secret).
2
Create the tables (SQL editor): leads, clients, client_leads, prospects, intakes, tickets, reviews, marketing_assets, client_profiles, pageviews, client_errors. Every migration ever run is saved under Supabase → Database → Migrations — that's the exact schema, replayable.
3
Turn on row-level security on every table with the three policy shapes from §4. This is the entire security model — the front-end is untrusted by design.
4
Build pages as single HTML files. Inline CSS, inline JS, supabase-js from CDN, talking straight to the database with the anon key. No build step, no framework. Auth = supabase.auth.signInWithPassword.
5
Code on GitHub, connect Vercel. Vercel → New project → import repo → every push auto-deploys. Point the domain at it.
6
Stand up n8n (Hostinger VPS template or any Docker host). Credentials: Gemini (AI), SMTP (email), Supabase service keys (database writes).
7
Wire forms → automations. Public pages insert into the database directly (fast, reliable) and/or POST to n8n webhooks (AI scoring + emails). Both patterns are used here on purpose.
8
Add the ops layer: uptime monitor, weekly backups, error logging, legal pages. All already in this repo — reuse them.
9
Multi-tenant (the sell-to-marketers phase): blueprint lives at hq/multi-tenant-blueprint.sql and hq/blueprints/nextjs/ — orgs table, per-org RLS, roles instead of a hardcoded admin email.

9 · Troubleshooting

SymptomCheck
Site downvercel.com → deployments (red = failed build → restore previous file via GitHub History). The uptime monitor emails you automatically.
Marketing Engine "Generation failed"n8n → is "TB — Marketing Engine AI" Active? → Executions tab shows the exact error. Usually the Gemini credential.
Emails not sendingn8n → Executions → open the red run. Usually the SMTP credential (App Password) or Gmail's ~500/day limit.
Client says CRM is emptyTheir clients.email must equal their login email, and clients.business must match the capture page's ?c= slug (spacing/case don't matter).
Lead didn't appearSupabase → Table editor → client_leads — in the database? If yes it's a display filter; if no, check the form page and the client_errors table.
Broke something editingGitHub → History on that file → restore previous version → commit.

10 · Developer handoff notes

To whoever inherits this: it's intentionally boring tech. Static HTML pages (one file per page, everything inline), Supabase as the only backend (Postgres + auth + storage; security enforced by RLS, not the client), n8n for anything scheduled or AI. Conventions: escape all user content with the local esc() before innerHTML; additive-only migrations; admin access = email check in RLS (a roles table is the planned upgrade); the anon key is public by design and service keys live only in n8n. Deploy = push to main, ~30s. n8n/README.md documents every workflow's credentials. Read hq/ARCHITECTURE.md, then this page's §4 and §8. There is no build step, no package.json, no node_modules — don't add one without a reason.
